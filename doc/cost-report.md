# DSF Cost Report

Shows token usage and cost per subagent and orchestrator after a run.

## Usage

```bash
# Current run (filtered on agent.log)
python3 .claude/hooks/cost-report.py

# All sessions (full JSONL history)
python3 .claude/hooks/cost-report.py --all
```

Requires that `agent.log` exists in `.claude/logs/`. Finds JSONL files automatically via git root.

## Columns

| Column | Explanation |
|---------|-----------|
| **Agent** | Who used tokens — subagent (from agent.log) or `orchestrator` (main conversation) |
| **Model** | Claude model that ran the agent |
| **Inp** | Input tokens — text sent TO Claude (instructions, file content, context) |
| **Out** | Output tokens — text generated BY Claude (code, analyses, responses) |
| **CacheW** | Cache Write — tokens written to prompt cache (pays 1.25× input, but only once) |
| **CacheR** | Cache Read — tokens read FROM cache (pays 0.1× input — the cheap repetition) |
| **Total** | Inp + Out + CacheW + CacheR |
| **Time** | Duration from START to DONE in agent.log |
| **Cost** | USD, based on Anthropic's pricing |

## How Correlation Works

All subagents and the orchestrator share one JSONL session file — Task agents are not separate processes. The script correlates each individual API call (timestamp in JSONL) against the agent window (START/DONE in agent.log) to distribute cost per agent.

Calls that fall outside all agent windows → `orchestrator`.

## Typical Pattern

The orchestrator dominates **CacheR** because the same long context (CLAUDE.md, plan document, file content) is read from cache in each new round. Subagents are cheap because they start with fresh context and run briefly.

**Why must the orchestrator re-read the context every round?**

LLMs are stateless — no memory between API calls. Each time the orchestrator sends a message to Claude, the full conversation history must be sent again: all previous messages, tool results, and file content. Claude "remembers" nothing from the previous call.

Prompt cache makes this cheap: the first time the context is sent it is written to cache (CacheW, 1.25× input). The next call hits the cache (CacheR, 0.1× input). After 25 agent rounds, the orchestrator carries 27 agent reports, plan updates, and file content — all sent from cache on every call.

Subagents always start fresh, without accumulated history — that is why they are cheap.

Example from a full compliance service run (sonnet-4-6):

```
Subagent (architect, dev, review, test):  $0.03–0.06 each
Orchestrator (main conversation):         $4–7
Total for full cycle (dotnet-native):     ~$5–7  / ~30 min
Total for full cycle (Docker):            ~$8–10 / ~55 min
```

## dotnet-native vs Docker

`dotnet test` and `dotnet run` are significantly faster than `docker build`:

| | dotnet-native | Docker |
|---|---|---|
| Build + test per phase | 5–15s | 1.5–3 min |
| Full run | ~30 min | ~55 min |
| Savings per cycle | | ~25 min / ~$2 |

Docker is slow because code changes invalidate the build layer — full recompilation on every `docker build`. `dotnet` uses incremental build with MSBuild cache.

**Recommendation:** Use dotnet-native (local SDK) for development environments. Docker only for CI and distribution.

## Avoiding Error Costs from Unknown Subagent Types

`subagent_type: "eval"` and `subagent_type: "architect"` do not exist in Claude Code. If used, they trigger an internal retry via the `[Plan]` agent — $1–2 per failed call, without any work being performed.

**Rule:** Always use `general-purpose` and inject the skill content directly into the prompt.
